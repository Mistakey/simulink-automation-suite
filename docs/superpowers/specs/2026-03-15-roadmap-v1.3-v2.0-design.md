### 1. Shared Layer Extraction

#### Target Structure

```
skills/
  __init__.py              # NEW ŌĆö required for skills._shared imports
  _shared/                 # No SKILL.md ŌåÆ plugin auto-discovery ignores it
    __init__.py
    session.py             # From sl_session.py: session discovery, connect, state
    validation.py          # From sl_core.py: validate_text_field, _validate_json_type, control char checks
    errors.py              # From sl_errors.py: make_error()
    json_io.py             # From sl_common.py: emit_json(), as_list(), JsonArgumentParser
  simulink_scan/
    (unchanged skill directory)
```

#### Why `_shared/` Is Safe

Plugin auto-discovery scans `skills/` for subdirectories containing `SKILL.md`.
`_shared/` has no `SKILL.md`, so it is invisible to the plugin system.
The underscore prefix further signals "internal library, not a skill."

#### Import Path Migration

```python
# Before (v1.2.x)
from .sl_common import emit_json
from .sl_errors import make_error
from .sl_session import connect_to_session

# After (v1.3.0)
from skills._shared.json_io import emit_json
from skills._shared.errors import make_error
from skills._shared.session import connect_to_session
```

#### `skills/__init__.py` Requirement

Currently `skills/__init__.py` does not exist. Python resolves `python -m skills.simulink_scan` via namespace packages, but explicit `skills._shared` imports require either:
- An explicit `skills/__init__.py` (recommended ŌĆö avoids ambiguity), or
- Reliance on PEP 420 implicit namespace packages

Decision: add an empty `skills/__init__.py` in v1.3.0.

#### Old Module Migration Strategy

After extracting shared code, the original files (`sl_common.py`, `sl_errors.py`, `sl_session.py`) in `simulink_scan/scripts/` are **deleted** ŌĆö not kept as re-export shims. Rationale:
- The plugin is a single installable unit; there are no external consumers of internal import paths.
- All tests update their import paths in the same commit.
- Re-export shims add maintenance burden and confuse future readers about the canonical location.
- Clean break is safe because the entire codebase is under our control.

#### Shared Layer Scope (Thin and Stable)

Only skill-agnostic, rarely-changing code goes into `_shared/`:

| Module | Contents | Lines (est.) |
|---|---|---|

### 3. `find` Action

#### Purpose

Search for blocks by name and/or type within a model or subsystem scope.
Fills the gap in current read-only workflow: agents can locate blocks without recursive scan + manual filtering.

#### Parameters (`_JSON_FIELD_TYPES` Entry)

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `model` | string | No | null | Target model (same resolution as `scan`) |
| `subsystem` | string | No | null | Narrow search scope to a subsystem |
| `name` | string | No | null | Name substring match (case-insensitive, internally `(?i).*name.*` regex) |
| `block_type` | string | No | null | BlockType exact match (e.g., `SubSystem`, `Gain`, `PID Controller`) |
| `session` | string | No | null | MATLAB session name |
| `max_results` | integer | No | 200 | Output clipping |
| `fields` | array | No | null | Field projection |

**Constraint:** At least one of `name` or `block_type` must be provided. Otherwise ŌåÆ `invalid_input`.

**Input hardening:** `name`, `block_type`, `model`, `subsystem`, `session` all pass through `validate_text_field()` (control chars, reserved chars, whitespace, length Ōēż 256).

**Regex safety:** The `name` value is user-supplied and interpolated into a MATLAB regex pattern. Apply `re.escape(name)` before interpolation to prevent regex injection (e.g., metacharacters like `.`, `*`, `(`, `)`, `[`, `]` in block names). This ensures the `name` parameter always performs literal substring matching, not arbitrary regex.

**`validate_args()` branch:** Add an `elif args.action == "find":` branch in `validate_args()` that:
- Validates `model`, `subsystem`, `name`, `block_type`, `session` via `validate_text_field()`
- Enforces at least one of `name` or `block_type` is present
- Validates `max_results` is a positive integer

#### MATLAB Implementation

```python
import re

# Escape user input to prevent regex injection
safe_name = re.escape(name)

# Name-only (case-insensitive literal substring)
eng.find_system(scope, 'RegExp', 'on', 'Name', f'(?i).*{safe_name}.*')

# Type-only
eng.find_system(scope, 'BlockType', block_type)

# Combined (AND semantics)
eng.find_system(scope, 'RegExp', 'on', 'Name', f'(?i).*{safe_name}.*', 'BlockType', block_type)
```

Search scope resolved via existing `resolve_scan_root_path()`.

**Search depth:** `find` always searches recursively within the resolved scope. MATLAB's `find_system` searches the full subtree by default. Use `subsystem` to narrow the search tree for large models.

#### Output Format

```json
{
  "model": "my_model",
  "scan_root": "my_model",
  "query": { "name": "PID", "block_type": null },
  "results": [
    { "path": "my_model/Controller/PID_speed", "name": "PID_speed", "type": "SubSystem", "parent": "my_model/Controller" },
    { "path": "my_model/Controller/PID_current", "name": "PID_current", "type": "SubSystem", "parent": "my_model/Controller" }
  ],
  "total_results": 2,
  "truncated": false
}
```

Projectable fields: `path`, `name`, `type`, `parent`.

#### Error Handling

- Model/session errors: reuse existing codes (`model_not_found`, `model_required`, `session_required`, `session_not_found`, `engine_unavailable`, `no_session`)
- Neither `name` nor `block_type` provided ŌåÆ `invalid_input`, `suggested_fix`: "Provide at least one of name or block_type."
- No results found ŌåÆ empty `results` array (not an error), `total_results: 0`

#### Recovery Routing (SKILL.md Addition)

```
- find returns empty results ŌåÆ broaden name pattern, try different block_type, or widen scope (remove subsystem constraint)
```

### 4. `set_param` Action

#### Parameters (`_JSON_FIELD_TYPES` Entry)

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `target` | string | Yes | ŌĆö | Full block path to modify |
| `param` | string | Yes | ŌĆö | Parameter name |
| `value` | string | Yes | ŌĆö | New parameter value (string form, MATLAB handles conversion) |
| `dry_run` | boolean | No | **true** | Preview mode ŌĆö show diff without writing |
| `model` | string | No | null | Model disambiguation |
| `session` | string | No | null | MATLAB session name |

**Input hardening:** All string fields pass through `validate_text_field()`.

**`value` type note:** The `value` field is always a string. MATLAB's `set_param` accepts string values and handles type conversion internally. Document this explicitly in SKILL.md so agents do not attempt to pass numeric or boolean JSON values.

#### MATLAB Implementation

```python
# 0. Validate target block exists (matches existing inspect/highlight pattern)
try:
    eng.get_param(target, 'Handle')
except Exception:
    return make_error('block_not_found', ...)

# 1. Validate parameter exists on this block
try:
    old_value = eng.get_param(target, param)
except Exception:
    return make_error('param_not_found', ...)

# 2. Dry-run: return diff, do NOT write
if dry_run:
    return { current_value, proposed_value, rollback }

# 3. Execute: write + read-back verification
eng.set_param(target, param, value)
verified_value = eng.get_param(target, param)
return { previous_value, new_value, verified, rollback }
```

#### Output Format

**Dry-run mode** (preview):

```json
{
  "action": "set_param",
  "dry_run": true,
  "target": "my_model/Controller/PID_speed",
  "param": "Kp",
  "current_value": "1.5",
  "proposed_value": "2.0",
  "rollback": {
    "action": "set_param",
    "target": "my_model/Controller/PID_speed",
    "param": "Kp",
    "value": "1.5",
    "dry_run": false
  }
}
```

**Execute mode** (actual write):

```json
{
  "action": "set_param",
  "dry_run": false,
  "target": "my_model/Controller/PID_speed",
  "param": "Kp",
  "previous_value": "1.5",
  "new_value": "2.0",
  "verified": true,
  "rollback": {
    "action": "set_param",
    "target": "my_model/Controller/PID_speed",
    "param": "Kp",
    "value": "1.5",
    "dry_run": false
  }
}
```

`rollback` is an executable JSON payload ŌĆö persist it before proceeding to enable one-command undo.

#### Error Codes

| Error Code | Scenario |
|---|---|
| Reused | `engine_unavailable`, `no_session`, `session_required`, `session_not_found`, `model_not_found`, `block_not_found`, `invalid_input`, `unknown_parameter`, `invalid_json`, `json_conflict` |
| `param_not_found` | NEW ŌĆö parameter name does not exist on the target block |
| `set_param_failed` | NEW ŌĆö MATLAB `set_param` call raised an error |

Error envelope shape unchanged: `{error, message, details, suggested_fix?}`.

#### Safety Model

1. **`dry_run` defaults to `true`** ŌĆö Agent must explicitly pass `dry_run: false` to write.
2. **Rollback payload in every response** ŌĆö Both dry-run and execute responses include a `rollback` field.
3. **Read-back verification** ŌĆö After `set_param`, the value is read back to confirm the write took effect.
4. **One parameter at a time** ŌĆö No batch operations in v2.0. Each invocation changes one parameter on one block.

### 5. Agent Workflow Pattern

Typical read-understand-preview-write-verify cycle across both skills:

```
simulink-scan  list_opened           ŌåÆ discover models
simulink-scan  scan --recursive      ŌåÆ understand topology
simulink-scan  find --name "PID"     ŌåÆ locate target block
simulink-scan  inspect               ŌåÆ read current parameters
simulink-edit  set_param (dry_run)   ŌåÆ preview proposed change
simulink-edit  set_param (execute)   ŌåÆ apply change
simulink-scan  inspect               ŌåÆ verify result
```

1. Add to `_JSON_FIELD_TYPES` in `sl_core.py`

6. Version bump
