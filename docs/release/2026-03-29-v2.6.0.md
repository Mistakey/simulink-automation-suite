# v2.6.0 Release Notes

## Summary

New `matlab_eval` action — execute arbitrary MATLAB code and return captured text output. This is the highest-priority feature in the backlog (F-001, P0), acting as a single generic entry point that unlocks workspace access, expression evaluation, and advanced MATLAB API calls.

## Highlights

- **`matlab_eval` action**: Executes arbitrary MATLAB code via `evalc`, returning captured command window output as text. Supports multi-line code, configurable timeout (default 30s), and output truncation at 50K characters for context window safety.
- **Unlocked capabilities**: With `matlab_eval`, agents can now perform workspace read/write (`evalin`/`assignin`), read simulation results, evaluate parameter expressions, query Bus Selector signals, inspect port handles, read MATLAB Function block source via Stateflow API, and access Data Dictionaries — all without dedicated actions.
- **Safety**: Operational tier with soft guardrails — code length limit (100K), output truncation (50K), async timeout to prevent runaway execution.

## Compatibility / Upgrade Notes

- All changes are additive. No breaking changes to existing action schemas or response shapes.
- New action: `matlab_eval` with fields `code` (required), `timeout` (optional, default 30), `session` (optional).
- New error codes: `eval_failed`, `eval_timeout`.
- Schema version bumped from `2.5` to `2.6`.

## Validation

- 367 unit tests passing (15 new for `matlab_eval`)
- Schema introspection verified: `matlab_eval` registered with correct fields
- Docs contract verified: recovery routing and handoff updated
- No live test report for this release (transport layer `evalc` + async timeout behavior requires live MATLAB verification)
